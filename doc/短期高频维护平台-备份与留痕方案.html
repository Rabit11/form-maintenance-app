SHORT-CYCLE · HIGH-FREQUENCY MAINTENANCE

短期高频维护汇总平台
备份与留痕轻量方案

本平台不是长期主数据系统，而是一轮轮「汇总统稿 / 分表填报 / 校验入库 / 再导出」的短期作业台。 风险主要来自频繁覆盖、误确认、并行维护冲突，而不是十年级合规仓储。 因此采用「战役式保护」：短保留、密检查点、可快速回退、留痕够问责即可。

平台特征 短期 · 高频 · 汇总维护 相对方案 对上一版企业长留存方案的场景裁剪版 版本 V1.0 · 2026-07-15

用得怎样

几天到数周内密集上传、在线改、拆分导出；结束后进入下一轮或归档冻结。

坏得怎样

整表替换抹掉上午成果；主管并发改错；导出的不是刚筛的那一版。

护到什么程度

当天可回退、本轮可追责、收尾可打包带走；不为「永久司法链」过度建设。

01平台特征与设计取向

特征含义对备份/留痕的影响
短期战役 按任务周期使用，非常年 OLTP 保留期对齐「本轮任务 + 缓冲周」，不做多年月冻结默认
高频写入 多次预校验、确认入库、分表改、再导出 强调操作前检查点，而不是只靠夜间日备
汇总为主 总表↔分表合并，产出交付 Excel/ZIP 保护「当前工作版」+「最近几个确认版」；交付包可顺带存一份
角色并行 总部与多位类型主管同时维护 留痕要能分清「谁改了哪类分表」；冲突以批次/时间线解释
运维轻 内网 Docker，人力有限 自动化优先、界面少而硬；避免双人审批、KMS 等重体系
一句话原则：宁可「一天内多拍几张快照、保留两三周」，也不要「一套重备份制度却挡不住今天下午的一次误替换」。

02与长留存方案的差异

上一版（偏企业合规）

  • 月冻结存 1–3 年
  • 哈希链 + WORM
  • 季演练、双人删档
  • 对象存储 / KMS

本版（短期高频）

  • 任务周期内密集检查点
  • 轻量校验和 + 异目录
  • 收尾打包演练即可
  • 本机第二路径足够

若后续某轮任务被指定为「正式归档依据」，再对该轮启用加长保留即可（按任务开关），不必全局一直重。

03检查点备份（战役式）

3.1 保护什么

  • 工作库快照：SQLite 一致性备份（Online Backup），含当前台账与导入批次状态；
  • 最近上传原件：与「待确认 / 已确认」批次绑定的 xlsx（避免只剩库、对不上文件）；
  • 任务标签：如「2026Q3预研汇总-第3轮」，所有检查点挂在同一任务 ID 下，方便整轮带走或整轮丢弃。

3.2 何时拍（比「每天一次」更重要）

触发优先级说明
确认入库 / 整表替换前 P0 必做 挡住「一次替换毁掉一上午」；失败应拦截或强告警
分表合并上传预校验通过后 P1 多主管并行时，留下合并前可回退点
总部手动「存一版」 P1 上午/下午节点或外出前随手拍
整点 / 每 2–4 小时（任务进行中） P2 无人触发时的安全网；任务结束后可关掉
任务收尾冻结 P0 打一个「Final」包：库 + 关键上传 + 导出样本哈希

3.3 保留多久（对齐短期）

类型保留用途
操作前检查点 最近 20～30 个，或 7～14 天 分钟～天级回退
每日末版本 本任务周期内每天 1 份，周期结束后再留 14～30 天 隔天找回
任务 Final 包 周期结束后 30～90 天（可配置） 对本轮汇总结果负责；到期提醒清理
不设默认「年备」。磁盘空间优先保证高频检查点不中断；过期自动淘汰，避免卷被旧库堆满拖慢平台。

3.4 恢复怎么做才「像维护平台」

  • 一键回退到上一检查点（旁路恢复预览 → 确认切换），面向总部，而不是只给运维敲命令；
  • 回退必须写 audit：谁在何时从哪一拍恢复；
  • 默认不挑战「跨任务恢复」；本任务内回退为主。

04留痕与短期存档

4.1 仍要回答的问题

在任务进行中与结束后两周内，能说清：

  • 这条记录是谁（工号）改的、改了哪些字段；
  • 这次入库依据哪次上传批次；
  • 谁导出了全量 / 某专项分表（防误传争论）。

4.2 热库：任务内「够用且快」

  • 字段 changelog、audit 任务期间全量在线，不做烦琐冷查询；
  • 界面继续用现有「变更明细 / 校验报告」；
  • demo「加载样本」允许清空业务表，但须：先自动打检查点，并在 audit 标明「演示重置」。

4.3 冷存：任务级打包，而不是永动哈希链

任务收尾或每日末，导出一份轻量「本轮留痕包」即可：

  • change_logs_YYYYMMDD.jsonl（或按任务 ID）；
  • audit_YYYYMMDD.jsonl；
  • manifest.json：行数、类型数、备份文件名、SHA-256；
  • 可选：当日出过的关键 Excel 文件名列表（不必重复存巨型包）。
不做默认区块链级哈希链。短期场景用「文件级 SHA-256 + 清单」足够防误改；真正要防内部恶意篡改时再对本轮 Final 包加签。

4.4 与筛选/导出的关系

本平台下载强依赖当前筛选与角色范围。建议在 audit 中固化： 导出文件名 + 筛选条件摘要 + 行数 + 工号， 这样「我要的类型 / 我下的类型」争议可用日志直接结案，而不靠事后回忆。

05一日作业节奏（推荐）

开工 → 确认当前任务标签；必要时从昨夜 Daily 恢复。

上午维护 → 各主管改分表 / 上传；合并前自动检查点。

午间 → 总部手动「存一版」或依赖整点任务。

下午确认入库 → 入库前强制快照 → 确认 → 抽查 changelog。

导出交付 → 按类型/筛选导出；audit 记清条件。

收工 → Daily 包；任务最后一天打 Final 包并冻结写权限（可选）。

06安全（够用即可）

  • 异目录：检查点写到宿主机 /backup/form-maintenance/{taskId}/，与容器 data 卷分离。
  • 权限：仅总部 / 超管可回退与下载备份；主管只产生留痕，不能删检查点。
  • 完整性：每份快照带 SHA-256；回退前校验。
  • 工号：写操作、确认入库、导出继续绑定六位工号（已有方向保持）。
  • 口令：演示阶段「密码=工号」可接受，但备份目录勿对全员开放目录浏览。
  • 不做：默认双人删除、默认异地多活、默认多年 WORM——除非单轮任务被升级为「正式归档轮」。

07落地阶段（贴合短期平台）

Phase 1 · 先挡住最大事故（建议优先）

确认入库 / 整表替换前自动检查点；最近 N 份快照列表 + 一键回退；失败告警。

Phase 2 · 跟上高频节奏

任务标签；进行中每 2–4 小时自动拍；手动「存一版」；Daily 包。

Phase 3 · 收尾可交代

任务 Final 包（库+留痕 JSONL+manifest）；导出 audit 带筛选摘要；到期清理提醒。

08验收标准（短期场景）

  • 总部误点「整表替换」后，10 分钟内能回到替换前检查点。
  • 任务进行的任意一天，能回答某个项目类型字段「上午谁改过」。
  • 一次导出争议可用 audit 中的筛选条件 + 行数当场核对。
  • 任务结束后，Final 包可单独拷贝存档；清理过期检查点不误删 Final。
  • 备份目录与业务 volume 分离：删容器卷不会同时带走全部检查点。
本方案是场景裁剪版：服务于短期频繁汇总维护。若个别轮次升级为长期依据，对该任务单独打开「加长保留 / 强化归档」开关即可。
