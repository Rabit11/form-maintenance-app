00现状与缺口

当前业务库位于 Docker 数据卷内的 SQLite（platform.db + WAL）， 并附带上传文件目录。系统已具备两类在线留痕，但尚不足以支撑「可恢复 + 可长期溯源 + 抗同损」。

类型 载体 现状特点 主要缺口
字段变更 transition_change_logs 行级 diff、操作人、时间 多为在线截断查询；demo/重置可能整表清空
行为审计 audit 登录、导入、导出、授权等 无独立归档；与业务库同盘同损
备份 Docker Volume 卷持久化，重启不丢 无定时冷备、无校验和、无异地副本、无恢复演练
核心风险：备份、归档与热库挤在同一存储域时，人为误删、卷损坏或恶意覆盖会导致「数据与证据同时消失」。

01设计目标

  • 可恢复：库损坏、误删、误确认入库后，能回到可信时间点。
  • 可溯源：任意字段变更与关键操作可回答「谁、何时、对哪条、改了什么、依据哪次导入」。
  • 抗篡改：冷备与 changelog 归档对超级管理员也尽量只增不改；读取留审计。
  • 可运维：自动调度、保留策略清晰、恢复可定期演练。

02数据库备份机制

2.1 备份对象（三层）

层级 内容 说明
L1 热库一致快照 platform.db 使用 SQLite Online Backup / .backup，禁止直接拷贝运行中的库文件（含 WAL 不一致风险）
L2 旁路资产 uploads/、导入原件、模板副本 与库中 uploads、导入批次记录对齐，支撑「入库依据哪份 Excel」
L3 配置 级联 JSON、脱敏后的环境清单、镜像/版本号 灾难后可复现运行环境
业务侧「导出 Excel / ZIP」仅作交付物，不能替代主备份。系统真相仍是数据库 + 上传原件。

2.2 备份节奏与保留

类型 频率 建议保留 用途
准实时 / 小时级 WAL checkpoint 后或关键写后触发 约 7 天 快速回退近期误操作
日全量 每日低峰（如 02:00） 30 天 常规恢复主路径
周全量 每周 12 周 中期回溯
月冻结 每月初 ≥ 1–3 年（按制度） 合规与长期存档

2.3 强制触发点

下列操作应触发备份（建议成功落盘或可靠入队后再视为业务完成；至少异步必达并失败告警）：

  • 总部「确认入库 / 整表替换」——替换前快照尤为关键；
  • 人员角色、工号、单位权限的批量变更前；
  • 超级管理员手工「立即备份」。

2.4 存放与安全

  • 异地 / 异卷：不得只保存在同一 Docker Volume；落到宿主机第二路径或对象存储。
  • 命名规范：{env}_{yyyyMMdd_HHmmss}_{imageOrGit}_{sha256前8}.db[.gz]
  • 完整性：同步写入 SHA-256 清单；恢复前强制校验。
  • 加密：静止数据磁盘加密或备份包口令 / KMS；传输走受限通道。
  • 访问控制：仅运维 / 超管可读；列表、下载、恢复全部写入 audit。
  • 密钥分离：备份密钥不进业务库。

2.5 恢复能力

  • 误删单行 / 误改字段 → 优先用 changelog + 业务规则回滚或比对恢复；
  • 误确认入库 → 恢复「确认前」快照到旁路库核对，再决定是否切入；
  • 全毁 → 最新日全量 (+ 必要时小时备份 / 归档增量)。
制度要求：每季度至少一次沙箱恢复演练，记录 RTO、失败点与改进项。

03Changelog 存档机制

原则：将热路径查询与冷归档存证拆分，避免无限膨胀压垮在线库，同时保证证据长期可用。

3.1 热库（在线）

  • transition_change_logs：保留约 90 天或最近 N 万条（取更严者）；
  • audit：同等窗口；登录 / 导出等高风险事件可酌情延长；
  • 支撑界面「变更明细 / 校验报告」的快速查询。

3.2 冷归档（只增）

超出窗口后滚动归档至独立目录或只读库；应用账号对归档目录无 DELETE 权限。建议单条结构：

archive_id / seq
event_type: field_change | audit | import_batch | export
occurred_at
actor_name / emp_no / role        ← 归档瞬间快照
object: identity_key / project_type / project_name / batch_id
action + diff_json                ← 字段旧值 / 新值
source_file / upload_id
client_meta: IP / UA（可选）
content_hash: SHA-256(本条)
prev_hash: 上一条哈希             ← 链式，防抽条篡改

导入批次、确认人、上传文件内容哈希应一并归档，形成「人—时—文件—字段」完整证据链。

3.3 写入纪律（溯源质量）

  • 写台账操作强制记录 工号 + 姓名；
  • diff_json 规范为字段码 / 旧值 / 新值；无实质变化不落库；
  • 禁止在无归档副本时执行清空 changelog / 重置演示库的破坏性操作；销毁须审批并产生审计事件；
  • 导出台账必须写 audit（谁、何时、导出了哪类包），支撑泄密追责。

3.4 查询与合规导出

  • 业务页：仅查热库近期记录；
  • 超管「溯源中心」：按项目 / 工号 / 时间 / 批次检索热库 + 冷归档（冷只读）；
  • 合规包：选定区间打包 changelog 归档 + 对应备份清单与哈希，供稽核使用。

04安全控制要点

  1. 职责分离：分表主管不可删备份 / 归档；超管可下发备份与只读调阅；物理删除建议后期引入第二人确认。
  2. 不可变偏好：归档目录 append-only 或 WORM；业务进程无删除权。
  3. 防同生命周期丢失：容器重建、compose down 不得清理冷备路径。
  4. 敏感字段策略：个人隐私可脱敏；项目名、经费等业务关键字段需明文才能真正溯源（靠权限收口）。
  5. 统一时钟：容器 Asia/Shanghai，归档时间统一 ISO-8601，避免跨机歧义。

05与现有模块衔接

能力 建议挂点
定时备份 宿主机 cron 或 backup sidecar，调用安全的 SQLite backup API
入库前快照 confirm 路径：确认前排队 pre-confirm 备份
Changelog 滚动 夜间任务：热库 → 冷归档，写入哈希链
超管界面 备份列表、沙箱恢复指引、归档检索（后续建设）
告警 连续备份失败、校验和失败、磁盘水位

演示环境可缩短保留期；正式环境严格执行上一节保留与异地策略。

06分阶段实施建议

Phase A · 1–2 周（价值最大）

日全量异地落盘 + SHA-256；确认入库前自动快照；禁止无归档清空 changelog。

Phase B · 随后

Changelog / audit 按月冷归档 + 哈希链；超管只读检索与合规打包导出。

Phase C · 加固

恢复演练剧本、保留策略配置化、备份加密、双人删档、监控告警看板。

07验收标准

  • 任意工作日可恢复到「昨日常规备份」点，RTO 可量化并记录。
  • 保留期内任意台账字段变更可还原「人 + 工号 + 时间 + 前后值 + 批次/文件」。
  • 备份损坏或缺档可在 24 小时内被发现（校验或监控）。
  • 归档抽检无法在不被发现的情况下改写历史（哈希链断裂即告警）。
本方案确认后，可继续输出「表结构示意 + 目录规范 + 接口清单」，或按 Phase A 直接进入实现。
